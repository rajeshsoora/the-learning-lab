# Profile Schema

The schema for `references/profile.md` — Vineel's per-domain calibration data, accumulated across sessions.

Read by Stage 1 (to pre-fill calibration fields) and Stage 2 (to size chunks).
Written by Stage 6 (after each session).

---

## File structure

`profile.md` is one markdown file with one `## Domain:` section per domain. New domains append; existing domains update in place (don't duplicate).

```markdown
# Vineel's Learning Profile

Last updated: YYYY-MM-DD

---

## Domain: [domain name, lowercase, e.g., "distributed systems"]

**Chunk size:** small / medium / large
**Sustain window:** N minutes (before quality degraded)
**Angles that worked:** [list]
**Angles that didn't work:** [list, with brief why]
**Prerequisites confirmed:** [list of concepts Vineel demonstrably owns in this domain]
**Known gaps:** [list, with date discovered]

**Sessions:**
- YYYY-MM-DD — Source: [name] — Artifact: [short description or link]
- YYYY-MM-DD — Source: [name] — Artifact: [short description or link]

**Notes:**
[Free-form: anything that would help future sessions in this domain — e.g., "prefers diagrams over prose", "fast on backend adjacencies, slow on theory"]

---

## Domain: [next domain]
...
```

## Field definitions

### Chunk size
How much new conceptual material Vineel can absorb in one loop before probe pass rate drops.

- **small** — 1–2 concepts per loop, often when domain is unfamiliar or theory-heavy
- **medium** — 3–5 concepts per loop, the default for adjacent-but-new domains
- **large** — 6+ concepts per loop, for domains where he has strong anchors

Calibrated by Stage 4 running calibration: if probes pass consistently, increase next session; if patches needed often, decrease.

### Sustain window
Minutes of focused session time before answer quality measurably degrades (slower probes, more hedging, more patches needed). Tracked per domain because cognitive load differs by topic.

- Backend systems: typically high (60–90 min)
- Theory-heavy domains (e.g., math, formal systems): typically lower (30–45 min)
- Brand-new fields: lowest (20–30 min)

Used by Stage 1 to flag overambitious budgets.

### Angles that worked / didn't work
Track per domain which projections (first-principles, mechanistic, comparative, applied, adversarial, historical) led to successful probes. This is the single most useful signal for future sessions.

Format:
- ✅ mechanistic — "explaining how it works step-by-step lands fast"
- ❌ historical — "too slow for this domain; just want the model not the timeline"

### Prerequisites confirmed
Concepts Vineel demonstrably owns in this domain (probed successfully at `use` or `teach` depth in past sessions). Lets Stage 3 Map step skip prerequisite mini-loops.

Format: bullet list with the concept name.

### Known gaps
Concepts that were parked in past sessions — explicitly not learned, with a reason. Useful for Stage 2 source selection ("this paper assumes X, which is in my known-gaps list — load it first").

Format: `- [concept] — parked YYYY-MM-DD — reason: [missing prereq / off-path / depth not needed yet]`

### Sessions
Append-only log. Date, source, artifact pointer.

### Notes
Free-form. Anything Vineel or Claude wants to remember about how learning works in this domain.

## Bootstrapping

`profile.md` doesn't exist initially. Create it on the first session that completes Stage 6. Use this header:

```markdown
# Vineel's Learning Profile

Generated by learning-framework skill. Read by Stages 1 and 2; written by Stage 6.
Last updated: [date]

---
```

Then append the first `## Domain:` block.

---

## When a domain doesn't fit cleanly

Some sessions cross domains (e.g., "RAG systems" overlaps with "backend" and "ML"). Two options:

1. Pick the dominant domain and note the crossover under Notes.
2. Create a hybrid domain entry (e.g., "ai/rag backend") if it's recurrent.

Don't proliferate domains for one-off sessions. A domain should accumulate ≥2 sessions before getting its own entry; before that, log under the closest existing domain.

---

## Editing by hand

This file is meant to be human-readable and human-editable. If Claude gets something wrong about Vineel (e.g., marks an angle as "didn't work" when it actually did in a particular context), he can edit it directly. Claude should re-read it at every Stage 1.
