Performance Idioms — Drilled, Not Just Shown

Module 08's Stage 08 shows each PySpark performance idiom once, anchored to the tuned ER pipeline. This page exists for a different job: drilling the same five idioms with fresh setups you haven't seen, two reps each, so a single correct read in the main module doesn't get mistaken for fluency. Nothing here is new conceptually — same pruning/sizing/broadcast/cache logic — this is purely about getting comfortable reading the code.

How to use this page: work top to bottom, idiom by idiom. Each gets a plain-language explanation, then two predict-then-check drills against a fresh toy setup. Each drill asks for the literal result of the idiom itself — how many partitions, how many MB, which side ships — not the higher-level "is this fast." Getting both right is a real signal; that's what references/code-fluency.md calls a "pass."
Why "logical result," not literal execution order. These are PySpark transformations — lazy and distributed. .cache() stores nothing until an action; .select/filter are pushed into the scan; repartition/coalesce act on a whole distributed dataset at once. Each reveal shows the logical outcome (final partition count, bytes read, what's materialized), not a literal row-by-row, line-by-line execution trace — Spark plans the whole DAG and runs it across the cluster, not statement by statement.

Idiom 1 — .select(cols) for column pruning

What it is, generally: picks a subset of columns. On a columnar format (Parquet/ORC) that's not just a projection — the file reader physically skips the unselected columns' bytes on disk. On a row-major format (CSV) it can't skip anything: the reader must read whole rows, so .select drops columns only after reading them all.

Where it shows up in the pipeline: .select("cust_id","name","email","phone","zip","status") right at the read — pruning 40 columns to 6.

columnsize on disk
id100 MB
name100 MB
email100 MB
notes100 MB
Drill 1 of 2: the table above is Parquet (400 MB total, 4 columns × 100 MB). You run df.select("id","name"). How many MB does the scan read off disk?
Reveal
select("id","name") on Parquet → reader touches only the id + name column-runs
→ 100 + 100 = 200 MB read (notes + email skipped on disk)
Drill 2 of 2 — the gotcha: same four columns, but the table is stored as CSV instead of Parquet. You run the identical df.select("id","name"). How many MB does the scan read off disk now?
Reveal
select("id","name") on CSV → CSV is row-major, columns interleaved per row
→ 400 MB read (all 4 columns), then id+name kept in memory
Same .select call, opposite cost. The pruning win lives in the file format, not the API. This is why "store it as Parquet" is a prerequisite for column pruning to mean anything.

Idiom 2 — repartition(n) vs coalesce(n)

What it is, generally: both set the number of partitions, but repartition(n) does a full shuffle and can set the count to anything — up or down. coalesce(n) only merges existing partitions with no shuffle, so it can only decrease the count; asking it for more than you have is a no-op.

Where it shows up: scored.coalesce(64) before the write — shrinking many post-shuffle partitions down to 64 files without paying a shuffle.

Drill 1 of 2: a DataFrame currently has 100 partitions. You call df.coalesce(25). How many partitions does the result have?
Reveal
100 partitions → coalesce(25) → merge down, no shuffle
→ 25 partitions
Decreasing is exactly what coalesce is for — it merges existing partitions locally, no network shuffle.
Drill 2 of 2 — the gotcha: a DataFrame currently has 12 partitions. You call df.coalesce(40), hoping for more parallelism. How many partitions does the result actually have?
Reveal
12 partitions → coalesce(40) → coalesce can only merge, never split
→ 12 partitions (the 40 is ignored)
To actually get 40 you'd need repartition(40), which splits and redistributes via a full shuffle. coalesce(40) on 12 silently leaves you at 12 — a classic "why is my job still not parallel" bug.

Idiom 3 — F.broadcast(df)

What it is, generally: wrapping a DataFrame in F.broadcast(...) tells Spark to copy that whole DataFrame to every executor and join locally — instead of shuffling both sides across the network to line up matching keys. It's only safe when the wrapped side is small (fits in each executor's memory).

Where it shows up: customers.join(F.broadcast(ref_zip), "zip") — the 5 MB zip lookup ships; the huge customers table doesn't move.

Drill 1 of 2: big is 500 GB, small is 8 MB. You run big.join(F.broadcast(small), "k"). Which DataFrame gets copied to every executor — big or small?
Reveal
F.broadcast(small) → the wrapped (small) side is shipped to every executor
→ small (the 8 MB side) is broadcast
You broadcast the small side precisely because copying 8 MB everywhere is cheap. Broadcasting big would be the disaster you're avoiding.
Drill 2 of 2: with that same big.join(F.broadcast(small), "k"), is the 500 GB big table shuffled across the network? (yes / no)
Reveal
small is on every executor → each big partition joins locally against it
→ no — big stays put, no shuffle of the 500 GB side
Skipping the shuffle of the big side is the entire point of a broadcast join. The big table is read once where it already lives and joined in place.

Idiom 4 — .cache() is lazy

What it is, generally: .cache() marks a DataFrame to be kept in memory after it's computed — but it computes nothing on its own. Like every transformation, it's lazy: the DataFrame is materialized and stored only when the next action (count, write, show, collect) forces the computation. From the second action onward, that action reads from the cache instead of recomputing.

Where it shows up: candidates.cache() before three branches — the first branch's action materializes it; branches two and three reuse it.

Drill 1 of 2: you run df.cache(), then df.count() (action 1), then df.filter(...).show() (action 2). On action 2, does Spark re-read df from the source? (yes / no)
Reveal
action 1 (count) computes df and fills the cache → action 2 reads the cache
→ no — action 2 reads from memory, source is not re-read
The first action pays the full cost and populates the cache. Every action after it skips the lineage and reads the stored result.
Drill 2 of 2 — the gotcha: you call df.cache() and then the program ends — no action ever runs on df. How many rows of df are stored in memory?
Reveal
cache() marks for caching, but nothing triggers materialization
→ 0 rows stored (cache is lazy; no action = nothing cached)
A .cache() with no following action is a no-op — people add it "to be safe" and are surprised it changed nothing. Materialization needs an action.

Idiom 5 — spark.conf.set("spark.sql.shuffle.partitions", n)

What it is, generally: a runtime config knob. Every wide transformation (groupBy, join, distinct, window) repartitions its shuffle output to this number. The default is 200, fixed and data-blind — it's set the same whether you shuffle 50 MB or 5 TB. (With AQE on, AQE coalesces this number down at runtime; these drills assume AQE off, to test the raw knob.)

Where it shows up: spark.conf.set("spark.sql.shuffle.partitions", "200") in the runtime-knobs block.

Drill 1 of 2: you run spark.conf.set("spark.sql.shuffle.partitions", "16") with AQE off, then a groupBy("k").count(). How many output partitions does that shuffle produce?
Reveal
shuffle.partitions = 16 → the groupBy shuffle writes 16 output partitions
→ 16 partitions
The knob directly sets the post-shuffle partition count — regardless of input size or row count.
Drill 2 of 2: in a fresh session you never set spark.sql.shuffle.partitions, AQE is off, and you run a join. How many output partitions does the join's shuffle produce by default?
Reveal
unset → falls back to the default value of spark.sql.shuffle.partitions
→ 200 partitions
This is the "200 trap": shuffle a tiny result and you still get 200 partitions — 200 tiny files. AQE's coalescePartitions exists specifically to trim this 200 down using the real output size.
