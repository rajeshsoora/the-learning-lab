ThresholdTuner Mechanics — Drilled, Not Just Shown

Module 05's Stage 07.2 shows you each PySpark idiom in ThresholdTuner once, anchored to that exact class. This page exists for a different job: drilling the same idioms with fresh data you haven't seen, two reps each, so a single correct guess in the main module doesn't get mistaken for fluency. Nothing here is new conceptually — same scoring/tuning logic as Module 05 — this is purely about getting comfortable reading the code.

How to use this page: work top to bottom, idiom by idiom. Each one gets a plain-language explanation, then two predict-then-check drills against a fresh toy dataset (not the P1–P8 set from Stage 07.2). Getting both right on an idiom is a real signal — that's what references/code-fluency.md calls a "pass," and two passes in two different places (this page counts as one, Stage 07.2 counts as another) is what moves an idiom toward "fluent."

Idiom 1 — F.when(cond, val).otherwise(val)

What it is, generally: PySpark's per-row if/else. It doesn't run once like a Python if — it evaluates cond for every row in the column and produces val or the otherwise-value, per row, all at once.

Where it shows up in ThresholdTuner: F.when(predicted & is_match, 1).otherwise(0) inside confusion_matrix — turning a boolean condition into a 0/1 column that F.sum can add up.

Drill 1 of 2: match_threshold = 0.75. For Q2 (composite=0.77, label=false_match): predicted = True, is_match = False. What does F.when(predicted & is_match, 1).otherwise(0) evaluate to for this row?
Reveal
predicted & is_match = True & False
→ False → F.when(...,1).otherwise(0) = 0
Drill 2 of 2: same threshold (0.75). For Q1 (composite=0.92, label=true_match): predicted = True, is_match = True. What does F.when(predicted & is_match, 1).otherwise(0) evaluate to?
Reveal
predicted & is_match = True & True
→ True → F.when(...,1).otherwise(0) = 1
Same expression, opposite row — this is the contrast that actually proves you can evaluate the condition, not just recall an answer.

Idiom 2 — boolean column-mask filter (& / ~ combined)

What it is, generally: predicted and is_match are each a column of True/False, one value per row. & is per-row AND, ~ is per-row NOT — for columns, not single booleans. Python's own and/not don't work here because there's no single value to test against.

Where it shows up: the fp and fn lines in confusion_matrix combine both — predicted & ~is_match and ~predicted & is_match.

Drill 1 of 2: For Q3 (composite=0.65, true_match), threshold=0.75: predicted=False, is_match=True. What does ~predicted & is_match evaluate to?
Reveal
~predicted & is_match = ~False & True = True & True
→ True
This is the fn condition — a real match the threshold missed. Q3 would count toward fn.
Drill 2 of 2: For Q4 (composite=0.50, false_match), threshold=0.75: predicted=False, is_match=False. What does predicted & ~is_match evaluate to?
Reveal
predicted & ~is_match = False & ~False = False & True
→ False
Q4 doesn't count toward fp here — predicted being False already rules it out, regardless of what ~is_match is.

Idiom 3 — Row attribute access after .first()

What it is, generally: .first() returns one Row object — think of it as a dict you can also dot-access. row.some_column reaches in by name, same as row["some_column"].

Where it shows up: cm.tp, cm.fp, cm.fn in pr_at, right after cm = self.confusion_matrix(...).

Drill 1 of 2: row = Row(precision=0.8, recall=0.6). What does row.recall evaluate to?
Reveal
row.recall
→ 0.6
Drill 2 of 2: row = Row(tp=5, fp=2, fn=1). What does row.tp evaluate to?
Reveal
row.tp
→ 5
Order in the Row constructor doesn't matter for dot-access — only the name does.

Idiom 4 — .cast("int") truncates, doesn't round

What it is, generally: converts a value's type. For floats → int, it chops the decimal off. It does not round to the nearest integer.

Where it shows up: (F.col(score_col) * bins).cast("int") in calibration_table, computing which bin a score falls into.

Drill 1 of 2: Q2's composite is 0.77. 0.77 * 10 = 7.7. What does .cast("int") produce for 7.7?
Reveal
7.7 → .cast("int")
→ 7 (not 8)
Drill 2 of 2 — the gotcha: a pair has composite 0.99. 0.99 * 10 = 9.9. What does .cast("int") produce for 9.9?
Reveal
9.9 → .cast("int")
→ 9 (not 10)
Even at 9.9 — a hair under 10 — it still truncates down to 9. This is the case where "round vs. truncate" actually changes the answer, which is exactly why it's worth drilling twice.

Idiom 5 — .groupBy().agg() then .orderBy()

What it is, generally: groupBy(key).agg(...) is the SQL GROUP BY equivalent — collapse every row sharing a key into one output row, computing aggregates per group. The output row order is not guaranteed — Spark groups across a distributed cluster, with no inherent "first group, second group." .orderBy(key) is what actually sorts it.

Where it shows up: calibration_table — group by score bin, then sort by bin so the printed table reads low-to-high.

Drill: three bins come out of groupBy("bin").agg(...) with no order promised: 0.9, 0.2, 0.5. After .orderBy("bin") (ascending, the default), which bin prints first?
Reveal
orderBy("bin"), ascending: 0.2, 0.5, 0.9
first = 0.2
Without .orderBy, those three rows could print in any order, and that order could even change between runs — grouping doesn't sort, sorting is its own separate step.
