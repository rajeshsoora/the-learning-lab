Spark Performance · Remaining Levers

Stage 00 · Overview

The Pipeline Works. Now Make It Cheap.

By Module 07 your entity-resolution pipeline is functionally complete — it standardizes, blocks, scores, classifies, tunes, and clusters into master entities. Every result it produces is correct. This module changes none of those results. It changes what they cost: the bytes read off disk, the data shuffled across the network, the memory each task burns, and the number of files you write. Same answers, a fraction of the cluster bill.

Module 08 doesn't add a stage — it tunes every stage you already built
Standardize→ Block→ Self-Join→ Score→ Classify→ Cluster→ Golden Record

Module 00 already gave you the three structural levers — shuffle, join strategy, and skew. Those decide the shape of the computation. This module is the remaining levers that decide how much each shape costs.

The five remaining levers

Lever 1 · read less
Column Pruning
Read only the columns the job actually uses. A 40-column table read for 6 columns wastes 85% of the scan. .select(...) early.
Lever 2 · read less
Predicate Pushdown
Push the WHERE into the scan so disk and the file format skip rows before Spark ever sees them. .filter(...) before the expensive op.
Lever 3 · right-size
Partition Sizing
Aim for ~128 MB per task. Too many tiny partitions = scheduler overhead and tiny files; too few giant ones = no parallelism and spill. repartition vs coalesce.
Lever 4 · fit in RAM
Spill & Memory
When a task's data won't fit execution memory, Spark spills to disk and crawls. Usually a partition-sizing problem in disguise — not a "buy more RAM" problem.
Lever 5 · reuse / auto-tune
Caching & AQE
.cache() a DataFrame reused by several branches. Let Adaptive Query Execution re-tune partition counts and skew at runtime so you hand-tune less.
How to read this module. Levers 1–2 make Spark read less. Lever 3 makes each task the right size. Lever 4 keeps each task in memory. Lever 5 reuses work and lets the engine auto-tune. Every lever is shown against your real MDM pipeline, and Stage 08 assembles them into one tuned read of that pipeline. There's a separate Code Companion that drills the PySpark idioms themselves.

Stage 01 · The motivating problem

"It Runs — Ship It." The Most Expensive Sentence in Data Engineering.

A correct pipeline and a cheap pipeline are two different things, and Spark will happily give you the first while quietly costing you ten times the second. The trap is that nothing fails. No error, no wrong answer — just a job that reads far more than it needs and a bill nobody questions because "it works."

Your ER pipeline, untuned

Your customers table in Parquet is wide and deep — 40 columns, 2 TB, billions of rows, most of them inactive accounts you never resolve. The first stage of entity resolution needs exactly six columns and only the active records. Here's what the naive read does versus what the job actually needs:

What the read doesColumnsRowsBytes scanned
Naive: spark.read.parquet(path)40 of 40all 5B2.0 TBwhat you pay
What ER stage 1 actually consumes6 of 400.6B active~180 GBwhat you need

Same correct candidate pairs come out either way. But the naive read moves ~11× the bytes off disk and across the cluster. On a pay-per-scan engine that's a literal 11× line item; on a fixed cluster it's 11× the wall-clock and 11× the chance of a spill or an OOM somewhere downstream.

Anchor
It's the difference between SELECT * and SELECT the six columns you need in a SQL query against a huge table — except Spark's default ergonomics make SELECT * the path of least resistance. You already know not to SELECT * a 40-column table in production. This module is that instinct, applied to every read, shuffle, and write in the pipeline.
The trap restated: performance bugs don't throw. A wrong join throws or returns garbage and you fix it immediately. An untuned scan returns the right answer and bills you forever. The whole skill of this module is seeing the cost that isn't yelling at you.

Stage 02 · Lever 1 — read less

Column Pruning — Stop Reading Columns You Throw Away

Parquet (and ORC) are columnar formats: each column is stored in its own contiguous run of bytes on disk, not interleaved row-by-row. That single fact is what makes column pruning real and not cosmetic — if you don't ask for a column, Spark physically never reads its bytes off disk.

Row-major vs column-major, in one picture

A CSV stores a row at a time: id,name,email,…,col40 then the next row. To get column 3 you must read every row's full width. Parquet stores all the ids together, then all the names, then all the emails. Ask for three columns and the reader seeks to exactly those three runs and skips the rest.

ReadFormatColumns touched on diskBytes
read.csv(path) then select 6CSV (row-major)all 40 — must read full rows2.0 TB
read.parquet(path) then select 6Parquet (columnar)6 — pruned at the file reader~300 GB
read.parquet(path).select(6)Parquet (columnar)6 — pruned, planner-confirmed~300 GB
Why two Parquet rows look identical. Spark's optimizer pushes the select down into the Parquet reader for you in many cases even if you select later in the query. Writing .select(...) early makes the pruning explicit and guaranteed rather than hoping the optimizer figured out which columns survive to the end. With CSV there is nothing to push down — the format can't seek to a column — so column pruning buys you almost nothing until you store as Parquet.

The wide-table trap in MDM

MDM source tables are notoriously wide — every system that ever touched a customer bolts on its own columns (loyalty flags, marketing consent, legacy IDs, audit timestamps). Entity resolution cares about a tiny slice: identity fields. Pruning to that slice is the single highest-leverage change on most ER pipelines, because the scan is the biggest single cost and you're discarding 80%+ of every row.

column pruning — the six identity fields

# 40-column table; ER needs 6. Select at the read, not after the join.
customers = (
    spark.read.parquet("s3://mdm/customers")
         .select("cust_id", "name", "email", "phone", "zip", "status")
)
# Parquet reader now touches 6 column-runs on disk, skips the other 34.
Don't prune away a column you need two stages later. If clustering's golden-record step needs updated_at to pick the latest record (Module 03's window function), it has to survive the select. Prune to "everything ER touches anywhere downstream," not "everything stage 1 touches." Pruning too aggressively is a correctness bug; pruning to the real footprint is free money.

Stage 03 · Lever 2 — read less

Predicate Pushdown — Filter at the Disk, Not in the Engine

Column pruning skips columns. Predicate pushdown skips rows — and the magic is where it skips them. A pushed-down filter is evaluated by the file reader (and sometimes the storage layer) before the rows are ever handed to Spark. A filter that isn't pushed down reads every row into the engine and throws most away after the fact.

Two places a filter can live

✗ Filter that stayed in the engine
Read everything, discard late

Spark reads all 5B rows into the executors, runs the self-join and scoring across all of them, then applies status == "active". You paid the scan, the shuffle, and the join on 5B rows to keep 0.6B.

df.join(other).filter(col("status")=="active")

✓ Filter pushed into the scan
Skip rows at the reader

The status == "active" predicate is handed to the Parquet reader. Row groups whose status min/max can't contain "active" are skipped on disk. Only 0.6B rows ever enter the engine, so the join shuffles 0.6B, not 5B.

df.filter(col("status")=="active").join(other)

The optimizer reorders many filters for you, but two things block pushdown and put the filter back in the engine: a filter expressed after a non-trivial transform the optimizer can't see through (a UDF, an explode, a join on the filtered column), and a predicate the file format can't evaluate (a regex on a column with no stored statistics). When pushdown is blocked, you eat the full scan.

Partition pruning — the strongest form

If the table is physically partitioned on disk by a column — say .../ingest_date=2026-06-18/... — then a filter on that column skips whole directories without opening a single file. This is partition pruning, and it's the difference between scanning one day and scanning three years.

FilterMechanismWhat gets skippedStrength
ingest_date == '2026-06-18' (partition col)Partition pruningentire date directoriesstrongest
status == 'active' (Parquet col)Row-group min/maxrow groups that can't matchgood
regexp_like(name, '...')none — runs in enginenothing; full scannone

Verify it actually happened

Never assume pushdown — confirm it. .explain() prints the physical plan, and you look for the PushedFilters line under the Parquet scan. If your filter is listed there, the reader is doing the skipping. If it shows up only as a later Filter node, it stayed in the engine and you have a problem to chase.

confirm pushdown with explain()

customers.filter(F.col("status") == "active").explain()

# Look for this line under the FileScan parquet node:
#   PushedFilters: [IsNotNull(status), EqualTo(status,active)]
# If 'active' appears there → the reader skips non-active row groups on disk.
# If you only see a separate '+- Filter (status = active)' node → NOT pushed.

Stage 04 · Lever 3 — right-size each task

Partition Sizing — The Goldilocks Number Is ~128 MB

A Spark partition is the unit of parallel work: one partition = one task = one core's worth of data at a time. Get the size wrong in either direction and you pay. Too many tiny partitions and the scheduler spends more time launching tasks than doing work — and your writes explode into thousands of tiny files. Too few giant partitions and most of your cluster sits idle while a handful of overloaded tasks spill to disk.

Anchor — the desks again
From Module 00: a partition is a desk, a task is a person working that desk. 2 000 desks each with one sheet of paper means people spend all day standing up and sitting down (scheduler overhead). Four desks each buried under a 500-page stack means four people work while everyone else watches, and those four run out of desk space and start stacking paper on the floor (spill). You want every desk holding a comfortable, similar-sized stack — about 128 MB.

The two tools — and why they're not interchangeable

You change partition count with repartition(n) or coalesce(n). They look similar and behave very differently:

repartition(n)coalesce(n)
Shuffle?full shuffle — moves all datano shuffle — merges existing partitions
Can increase count?yes — any nno — only decrease
Result balanceeven, uniform sizescan be uneven (merged blindly)
Costexpensive (network)cheap (local merge)
Use whengoing up, or fixing skew/uneven sizesgoing down after a heavy filter

The canonical case: you read 2 TB into 16 000 partitions, then filter to the 0.6B active rows — now most of those partitions are nearly empty but you still pay one task each. coalesce(800) merges them down with no shuffle. Going the other way — you have 8 fat partitions and need parallelism — only repartition(200) can create partitions, and it costs a shuffle to do it.

The 200 default that bites everyone

Every shuffle (groupBy, join, distinct, window) repartitions its output to spark.sql.shuffle.partitions, which defaults to 200. That number is a fixed constant that knows nothing about your data. Shuffle 10 GB and 200 is roughly right (~50 MB each). Shuffle 50 MB through a small lookup join and you've just created 200 partitions of 0.25 MB each — 200 tiny tasks and, if you write them, 200 tiny files.

The tiny-files problem compounds downstream. Those 200 micro-files don't just waste a write — every job that later reads that output pays an open-and-close per file, and object stores like S3 throttle on request count, not bytes. A pipeline that writes thousands of tiny files per run quietly degrades every consumer of that data. Stage 06's AQE fixes the 200 default automatically; until then it's a manual knob you must set per workload.

Stage 05 · Lever 4 — the climax

Spill & Memory — "Just Add More RAM" Is the Wrong Reflex

This is the stage that separates people who tune Spark from people who throw money at it. When a job slows to a crawl with "spilling to disk" in the logs, the instinct is to bump executor memory. Sometimes that's right. Usually it's treating the symptom and missing that the real disease is a partition that's too big.

What "spill" actually is

An executor's memory is split into regions; the execution region is the scratch space a task uses to build hash tables for joins and aggregations and to sort. When a single task's working set doesn't fit that region, Spark doesn't crash — it writes the overflow to local disk and reads it back, repeatedly. That's a spill. It's correct, and it's slow, because disk is orders of magnitude slower than RAM and you're now round-tripping through it mid-operation.

Spill ≠ OOM. A spill is Spark gracefully degrading to disk to finish the job. An OutOfMemory error is a task asking for more memory than exists and the executor dying. Spill makes you slow; OOM makes you fail. They have overlapping causes but they are not the same event, and "add memory" helps OOM far more reliably than it helps spill.

The red herring, head-on

✗ Red herring — total memory
"It spilled, so the cluster needs more RAM"

You double executor memory. The spill shrinks a little, the bill doubles, and the next slightly-bigger input spills again. You've raised the ceiling, but the thing hitting the ceiling is one task's slice of data, and that slice grows with your data.

Spill is measured per task, against the memory one core gets.

✓ What actually decides it
Data-per-task vs memory-per-task

Spill happens when one partition is too big for one task's memory budget. Cut the partition size — repartition to more partitions, or fix the skew that made one partition huge — and each task's working set drops below the region it already has. No new RAM purchased.

More partitions → less data per task → no spill, at the same total memory.

This is why partition sizing (Stage 04) and spill (here) are the same lever viewed from two ends. A spill is almost always your cluster telling you a partition is too fat. The fix order is: (1) more/smaller partitions, (2) check for skew making one partition fat — Module 00's salting — and only then (3) more memory, when the per-task working set genuinely can't be made smaller.

Caching — the other memory lever, with its own trap

Storage memory (the other region) is where .cache() / .persist() keep a DataFrame so later actions reuse it instead of recomputing from source. The win is real when a DataFrame feeds several downstream branches: compute it once, read it from memory three times.

✓ Cache earns its keep
Reused by multiple actions

Your scored candidate set feeds the review queue, the metrics report, and the cluster step. Without a cache, Spark recomputes the whole scan-block-score chain three times. candidates.cache() pays for itself on the second use.

✗ "Cache everything" anti-pattern
Used once, or too big to hold

Caching a DataFrame used by a single action just burns storage memory for nothing — and worse, it can evict other cached data or squeeze the execution region, causing the very spill you were trying to avoid. Cache deliberately, not reflexively.

Cache is lazy. df.cache() marks the DataFrame for caching but materializes nothing until the next action forces it. The first action computes and stores; only from the second action on do you read from memory. Calling .cache() and never triggering an action caches exactly nothing.

Stage 06 · Lever 5 — let the engine tune itself

Adaptive Query Execution — Tuning That Happens at Runtime

Everything so far is something you decide before the job runs, on numbers you guessed (200 partitions? 128 MB?). The problem is that the right numbers depend on data the planner can't see until the shuffle actually produces it. Adaptive Query Execution (AQE), on by default in modern Spark, re-optimizes the plan mid-flight using the real sizes it observes — turning several of your manual knobs into automatic ones.

Three things AQE does for you

AQE feature 1
Coalesce shuffle partitions
After a shuffle, AQE sees the real output size and merges the fixed 200 down to a sensible count of ~target-sized partitions. The "200 default" trap from Stage 04 — fixed automatically.
AQE feature 2
Skew-join handling
Detects a partition that's far bigger than its siblings (one hot blocking key) and splits it into sub-partitions so it doesn't bottleneck the join. Module 00's skew, handled at runtime.
AQE feature 3
Dynamic join switch
If a side of a join turns out small enough after filtering, AQE switches a planned sort-merge join to a broadcast join on the fly — no manual broadcast() hint needed.

What AQE handles vs what stays on you

AQE is powerful but it only re-optimizes shuffle-stage decisions, because that's where it gets real size statistics. It can't undo a wasteful read — it never sees the columns and rows you should have pruned, because by then the scan already happened.

LeverAQE handles it?Why
Column pruningno — youhappens at the scan, before any shuffle stats exist
Predicate pushdownno — youa scan-time decision; AQE works post-shuffle
Shuffle partition countyes — AQEcoalesces 200 → right-size using real output size
Skew in a joinyes — AQEsplits the hot partition at runtime
Broadcast vs sort-mergeoften — AQEswitches if a side is small post-filter; explicit hint still wins for certainty
Caching decisionsno — youAQE doesn't know your DAG's reuse pattern
Initial read partition sizeno — youset by file/block sizes at read, pre-shuffle
The mental split: AQE owns the shuffle (partition counts, skew, late broadcast). You own the read (pruning, pushdown, initial sizing) and the DAG (what to cache). Turn AQE on and stop hand-setting shuffle.partitions per job — but don't expect it to save you from a SELECT *.

turn AQE on (modern Spark has these on by default — set them explicitly to be sure)

spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

Stage 07 · Decision criteria

Symptom → Lever — A Diagnostic You Can Run From the Spark UI

Performance tuning isn't applying all levers everywhere — it's reading a symptom and reaching for the one lever that addresses it. Each of these starts from something you can actually see in the Spark UI or the query plan, and points at the fix. Run them top to bottom.

Symptom 1
The scan reads far more bytes than the job uses — input size dwarfs the output.
→ Column pruning + predicate pushdown. Select only needed columns at the read; filter early and confirm PushedFilters in .explain(). Check the table is Parquet/ORC, not CSV. This is almost always the biggest single win.
Symptom 2
Thousands of tiny tasks, or the write produces thousands of tiny files.
→ Coalesce (or AQE coalesce). Too many partitions for the data volume — usually after a heavy filter or the 200-default shuffle. coalesce(n) down with no shuffle; or turn on AQE's coalescePartitions and let it pick.
Symptom 3
Tasks spilling to disk; a few tasks run far longer than the rest.
→ Smaller partitions first, check skew, then memory. Spill = a partition too big for one task. repartition to more partitions; if only one task is slow it's skew (salt it — Module 00, or AQE skewJoin). Add executor memory only when the per-task working set genuinely can't shrink.
Symptom 4
The same DataFrame is recomputed in multiple branches of the job.
→ Cache / persist. If a DataFrame feeds two or more actions, .cache() it so the scan-and-transform runs once. Only when it's genuinely reused and fits memory — caching a single-use or oversized DataFrame backfires.
Symptom 5
A join shuffles a huge table against a tiny lookup table.
→ Broadcast the small side. F.broadcast(small_df) ships the lookup to every executor and skips the shuffle of the big table entirely. AQE may do this automatically post-filter, but an explicit hint guarantees it. (Module 00's broadcast join, applied as a perf lever.)
One symptom, not all five. The mistake is reaching for memory because that's the knob you know. Read the actual symptom — bytes scanned, task count, spill metric, recompute, shuffle size — and the symptom names the lever. Most "Spark is slow" tickets are Symptom 1 in disguise.

Stage 08 · The production pattern

All Five Levers, on One ER Pipeline

Here is the entity-resolution read from Stage 01 — the one that naively scanned 2 TB — rewritten with every lever applied. Same candidate pairs out the other end; a fraction of the cost. Read it top to bottom; each highlighted line is one lever you just learned.

tuned_er_read.py — the pipeline with all five levers

from pyspark.sql import functions as F

# ── 1. Runtime knobs: let AQE re-tune shuffles for you ──
spark.conf.set("spark.sql.adaptive.enabled", "true")              # AQE on
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# ── 2. Read less: column pruning + predicate pushdown ──
customers = (
    spark.read.parquet("s3://mdm/customers")
         .select("cust_id","name","email","phone","zip","status")  # prune 40→6
         .filter(F.col("status") == "active")                # pushed into scan
)

# ── 3. Broadcast the small lookup — no shuffle of the big table ──
ref_zip = spark.read.parquet("s3://mdm/zip_lookup")            # ~5 MB
candidates = customers.join(F.broadcast(ref_zip), "zip", "left")  # ship the 5 MB side

# ── 4. Cache: candidates feed THREE downstream branches ──
candidates.cache()                                              # compute once, reuse 3×
scored   = score_pairs(candidates)        # branch 1
review_q = build_review_queue(scored)    # branch 2
metrics  = compute_metrics(scored)       # branch 3

# ── 5. Right-size the write: avoid the tiny-files explosion ──
scored.coalesce(64).write.mode("overwrite").parquet("s3://mdm/scored")  # 64 files, not 200

# ── 6. Verify the read actually pruned + pushed down ──
candidates.explain()    # expect PushedFilters:[EqualTo(status,active)] + 6 cols

The Code Companion — Stage 08.1 drills the same five idioms (.select, repartition vs coalesce, F.broadcast, .cache, spark.conf.set) with fresh examples and multiple predict-then-check reps, separate from this walkthrough. Optional — this stage stands on its own.
Order matters here. The select and filter come before the join and cache on purpose — prune and filter first so everything downstream operates on the smaller dataset. Cache after the expensive read-prune-filter-join chain so what's stored is the reusable result, not the raw scan. Reorder these and you cache the wrong thing or push the filter past where it can be pushed down.

Honest Gaps
What this module didn't cover
Bucketing & pre-sorted tables: writing data bucketed on the join key so repeated joins skip the shuffle entirely — a write-time investment for read-time speed, not covered here.

The unified memory model in detail: the execution/storage split was described conceptually; the actual spark.memory.fraction / storageFraction knobs and how borrowing works between regions weren't drilled.

Persistence levels: MEMORY_ONLY vs MEMORY_AND_DISK vs serialized/off-heap variants of .persist() were collapsed into ".cache()".
