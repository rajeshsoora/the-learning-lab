Module 01 · Complex Types

Arrays vs Structs vs Maps

Quick-reference decision card for your MDM pipeline. The type call is always about the key set — never the value type.

The Golden Rule
Fixed named parts → Struct · Variable count of same-type things → Array · Open-ended growing key set → Map
The value type is a red herring. Ask about the keys.

Decision Tree

Q1
Is the key set fixed, named, and known at schema-design time?
YES → Struct NO → Q2
Q2
Is it a variable-count collection of the same type of thing (no named parts)?
YES → Array NO → Q3
Q3
Open-ended key set, all values same type, keys unknown at schema time?
YES → Map NO → reconsider model

Type Reference

Array
Variable-count, same-type items
📍 Signal: "could be 0, 1, or many of the same thing"
phones: array<string>
["555-1234", "555-9000"]
Key ops: explode · explode_outer · filter · transform · size · array_contains
Struct
Fixed named parts — a contract
📍 Signal: "has a street, a city, a zip" — each field has a name and meaning
address: struct<street,city,zip,country>
{street:"1 Main St", city:"NY", zip:"10001"}
Key ops: dot access (address.city) · struct() rebuild · withColumn on nested field
Map
Open-ended key set, same-type values
📍 Signal: "keys grow as we add signals/plugins" — can't enumerate at schema time
signal_scores: map<string,double>
{"name_match":0.9, "email_match":0.7}
Key ops: col["key"] · map_keys · map_values · explode (→ key, value rows)
