Window Functions

The moment in every MDM pipeline where you need to ask "compared to the other records in this group, where does this one rank?" — that's a window function. This module explains why they exist, how they work, and exactly where they appear in your deduplication and golden record logic.

Where window functions appear in your pipeline
Ingest→ Standardize→ Block→ Self-Join→ Score→ Dedup / Rank ← window functions live here→ Golden Record

The Real Problem This Solves

After scoring, you have a table of customer records — but some are duplicates. You need to pick one winner per cluster. That means ranking records within each cluster and keeping only rank = 1.

Here's the trap: you might reach for groupBy().max(). It seems right — group by cluster, take the record with the highest score. But groupBy collapses rows into one. You lose the original row. You can't keep the full record, you only get the aggregated value.

Window functions solve this exactly: they let you compute a value that looks at a group of rows — without collapsing them. Every row keeps its full data, and gets a new column added with the result.

Stage 01

What is a Window?

This is the hardest concept in the module. Everything else builds on it. Read this section carefully before moving on.

The Problem with groupBy

Imagine you have 4 customer records from the same cluster (they're probably the same person). You want to find the one with the highest confidence score. Your first instinct might be groupBy("cluster_id").max("score").

Here's what that does — and why it fails for MDM:

❌ groupBy — collapses rows, loses the record
df.groupBy("cluster_id")\ .max("score")
Output — you get this:
cluster_idmax(score)
C10.94
C20.88
⚠ You have the score but lost the name, email, phone — the whole record is gone. You can't build a golden record from just a score.
✓ Window function — adds a column, keeps every row
w = Window.partitionBy("cluster_id")\ .orderBy(F.desc("score")) df.withColumn("rank", F.row_number().over(w))
Output — you get this:
cluster_idnamescorerank
C1John Smith0.941
C1Jon Smyth0.812
C2Priya Patel0.881
C2P. Patel0.722
✓ Every row is still there. Each got a rank column. Filter where rank=1 to get your golden records.

The Analogy — A Sliding Frame Over Sorted Rows

Analogy
Imagine you've sorted all your customer records into piles on a desk — one pile per cluster. Now you pick up each record one at a time, but instead of looking at it in isolation, you look at it in the context of its pile: "this record is #1 in its pile, the one below it is #2," etc.

The window is that pile. The window function is the question you ask while holding each record: "What position am I in my pile?" or "What was the score of the record above me?"

Crucially — you never merge the pile into one card. Every card comes back to the table with a new sticky note on it saying its answer.

The Three Parts of Every Window Spec

Every window function in PySpark needs a window specification — a description of how to form the frame. It has three parts:

Part 1
PARTITION BY
Which column defines the "pile"? In MDM: cluster_id. Records in the same cluster form one window. Records in different clusters never see each other.
Part 2
ORDER BY
Within the pile, how are records sorted? In MDM: by score DESC — so rank 1 = highest score. The order determines what "above" and "below" mean for LAG/LEAD.
Part 3 (optional)
ROWS/RANGE BETWEEN
How many rows to include in the frame. Default for most functions: all rows in the partition. You'll see this in running aggregates. Most MDM use cases don't need to set this explicitly.

Plain Python Analog — Before the Spark Syntax

Here's what a window function is doing, in pure Python, before any Spark API:

Pure Python — the idea behind a window function
# Imagine your records are just a list of dicts records = [ {"id": "R1", "cluster": "C1", "name": "John Smith", "score": 0.94}, {"id": "R2", "cluster": "C1", "name": "Jon Smyth", "score": 0.81}, {"id": "R3", "cluster": "C2", "name": "Priya Patel", "score": 0.88}, {"id": "R4", "cluster": "C2", "name": "P. Patel", "score": 0.72}, ] # Step 1: group by cluster — this is PARTITION BY from itertools import groupby clusters = {} for r in records: clusters.setdefault(r["cluster"], []).append(r) # clusters = {"C1": [R1, R2], "C2": [R3, R4]} # Step 2: sort each group by score desc — this is ORDER BY score DESC for cluster_id, group in clusters.items(): group.sort(key=lambda r: r["score"], reverse=True) # Step 3: add a rank to each record — this is ROW_NUMBER() # Notice: every record stays! We're just adding a new field. for cluster_id, group in clusters.items(): for rank, record in enumerate(group, start=1): record["rank"] = rank # ← new column added, record unchanged # Result: every record now has a rank field # R1 → {"id":"R1", "cluster":"C1", "name":"John Smith", "score":0.94, "rank":1} # R2 → {"id":"R2", "cluster":"C1", "name":"Jon Smyth", "score":0.81, "rank":2} # R3 → {"id":"R3", "cluster":"C2", "name":"Priya Patel","score":0.88, "rank":1} # R4 → {"id":"R4", "cluster":"C2", "name":"P. Patel", "score":0.72, "rank":2} # Step 4: filter rank == 1 — this is your golden record selection golden = [r for r in records if r["rank"] == 1] # → [{"id":"R1",...}, {"id":"R3",...}] ← one per cluster, full records kept
Key insight: PySpark's window functions do exactly this — but across a distributed cluster with millions of rows. The Window.partitionBy(...).orderBy(...) spec is just a way of telling Spark: "group these, sort them like this, then run the function." The Python loop above is the mental model. The Spark API is the production implementation.

Stage 02

PARTITION BY — Grouping Without Collapsing

PARTITION BY is the most important part of a window spec. It decides which rows can "see" each other. Rows in the same partition are in the same window. Rows in different partitions are completely invisible to each other.

The PySpark Syntax — Line by Line

Here's how you define a window spec in PySpark. Read each line as a direct translation of the plain-English idea:

Window spec — what each line means
from pyspark.sql import Window
Why this line: Window is not in F (pyspark.sql.functions). It's a separate import. You need it to define the partition/order spec before calling functions like row_number().
w = Window.partitionBy("cluster_id")
w now means: "look only at rows that share the same cluster_id value"
Why this line: This is the "pile" definition. Every row with cluster_id = "C1" will form one window. Every row with cluster_id = "C2" forms another. They never mix.
w = w.orderBy(F.desc("score"))
w now means: "within each cluster, sort rows from highest score to lowest"
Why this line: The sort order determines rank 1 vs rank 2. We want the best-scoring record to be rank 1, so we sort descending. If you forgot this, ranks would be assigned in arbitrary order — non-deterministic results.
df.withColumn("rank", F.row_number().over(w))
For each row: compute row_number() using window w, store the result in a new column called "rank". The original row is unchanged.
Why .over(w): row_number() on its own means nothing — it needs to know "row number within what?" The .over(w) tells it which window spec to use. Without it, Spark throws an error.
PySpark — defining and using a window spec
from pyspark.sql import Window # separate import — not in F from pyspark.sql import functions as F # Define the window: group by cluster, best score first w = Window\ .partitionBy("cluster_id")\ # ← which rows are in the same "pile" .orderBy(F.desc("score")) # ← sort within the pile, best first # Add a rank column — row_number() counts from 1 within each partition df = df.withColumn("rank", F.row_number().over(w)) # ↑↑↑↑↑↑↑ # .over(w) connects the function # to your window spec — required # Now filter: keep only rank 1 — one record per cluster golden_records = df.filter(F.col("rank") == 1)

Stage 03

ROW_NUMBER — Picking the Winner

ROW_NUMBER is the most used window function in MDM. It assigns a unique integer — 1, 2, 3 — to each row within its partition, based on the ORDER BY you defined. Rank 1 is your winner. This is how you pick the golden record from a cluster of duplicates.

The Full Dedup Code — With Debug Trace

Execution trace — what Spark is actually doing
w = Window.partitionBy("cluster_id").orderBy(F.desc("score"))
w = a specification object. No data touched yet. This is just a recipe.
Why define it separately: You'll reuse this spec for multiple window functions (row_number, lag, running count). Defining it once and referencing w keeps the code DRY.
df = df.withColumn("rank", F.row_number().over(w))
df now has one extra column "rank". All other columns unchanged. Row count unchanged (still 6 rows).
Why withColumn and not a new variable: withColumn returns a new DataFrame with the column added — it doesn't modify df in-place. Assigning back to df just overwrites the variable reference. The original data is immutable in Spark.
golden = df.filter(F.col("rank") == 1)
golden = 3 rows (R1, R3, R5). One per cluster. Each row has its full original data intact.
Why filter after withColumn and not before: You need the rank column to exist before you can filter on it. The window function must run first to produce the rank, then you filter.
PySpark — full dedup with ROW_NUMBER
from pyspark.sql import Window from pyspark.sql import functions as F # 1. Define the window spec # partitionBy = one pile per cluster # orderBy desc = best score gets rank 1 w = Window\ .partitionBy("cluster_id")\ .orderBy(F.desc("score")) # 2. Add rank column — does NOT collapse rows, does NOT change row count df = df.withColumn("rank", F.row_number().over(w)) # 3. Keep only rank 1 per cluster → your golden records golden = df.filter(F.col("rank") == 1)\ .drop("rank") # ← clean up helper column # Result: one row per cluster_id, full original data preserved # golden.count() == number of unique clusters
Common confusion — does filter(rank == 1) return just one cluster?

No — it returns one row from every cluster. Here's why:

When the window function runs, it processes the entire DataFrame at once. Every row in every cluster gets its rank column in a single pass. partitionBy doesn't mean "process one cluster at a time" — it means "when assigning ranks, each row only competes against rows in the same cluster."

So rank = 1 in C1 and rank = 1 in C2 are completely separate things. The rank counter resets to 1 for each partition — like running enumerate() fresh for each cluster pile. After the window function, C1's best row has rank 1, C2's best row also has rank 1, C3's best row also has rank 1.

filter(rank == 1) then just keeps every row where rank happens to be 1 — which is exactly one row per cluster. That's how you get R1 + R3 + R5 — all three clusters, one winner each.
The tie problem: R5 and R6 both have score 0.91. ROW_NUMBER will arbitrarily pick one as rank 1. This is non-deterministic — run it twice, get different results. In Stage 04 you'll see how to add a stable tie-breaker (like updated_at — more recent record wins).

Stage 04

RANK vs DENSE_RANK — Handling Ties

Three functions assign row numbers: ROW_NUMBER, RANK, and DENSE_RANK. They behave identically when there are no ties. When scores tie, each handles it differently. Knowing which to use prevents silent bugs where you expect one record per cluster but get zero or two.

Side-by-Side Output — Same Data, Three Functions

Using cluster C3 where R5 and R6 both score 0.91:

idnamescoreROW_NUMBERRANKDENSE_RANKKey difference
R5Sarah Jones0.91111All agree at the top
R6Sara Jones0.91211RANK+DENSE_RANK both give 1 — tied!
R7S. Jones0.65332RANK skips 2 (gap). DENSE_RANK doesn't.
ROW_NUMBER
Always unique. Never ties.
Even if scores are identical, one row gets 1 and the other gets 2. The choice is arbitrary. Use when you must have exactly one winner — add a stable tie-breaker to the ORDER BY to make it deterministic.
RANK
Ties get the same rank. Skips numbers.
Two rows tied at score 0.91 both get rank 1. The next row gets rank 3 (skips 2). filter(rank == 1) returns both tied rows. Use when you want all records tied for the top spot.
DENSE_RANK
Ties get the same rank. No gaps.
Same as RANK for tied rows, but the next distinct score gets rank 2 (not 3). Use when you care about "how many distinct score levels above this row" rather than "how many rows above this row."

MDM Recommendation — Make Ties Deterministic

For golden record selection, you almost always want ROW_NUMBER — but with a stable tie-breaker added to the ORDER BY so the result is the same every time you run the pipeline:

PySpark — stable tie-breaking with ROW_NUMBER
# ❌ Non-deterministic — if two rows have the same score, # rank 1 could be either one on different runs w_bad = Window.partitionBy("cluster_id")\ .orderBy(F.desc("score")) # ✓ Deterministic — same score? most-recently-updated record wins. # updated_at breaks any tie in a consistent, meaningful way. w_good = Window.partitionBy("cluster_id")\ .orderBy( F.desc("score"), # ← primary sort: highest score first F.desc("updated_at") # ← tie-breaker: most recent first ) df = df.withColumn("rank", F.row_number().over(w_good)) # Now: if R5 and R6 tie on score, the one updated more recently gets rank 1. # Deterministic. Repeatable. Auditable.

Stage 05

LAG & LEAD — Looking at Neighbouring Rows

LAG looks at the row before the current one in the window. LEAD looks at the row after. Both are used in MDM to detect score drift — how much did confidence drop between the best and second-best candidate? A big drop means a clear winner. A small drop means a borderline case needing review.

The Analogy — Looking at Your Neighbours in a Sorted Line

Analogy
Imagine the records in a cluster are standing in a line, sorted by score, best at the front. LAG(score, 1) means: "what score does the person directly behind me have?" LEAD(score, 1) means: "what score does the person directly in front of me have?"

The person at the front of the line (rank 1) has nobody in front — LEAD returns null. The person at the back has nobody behind — LAG returns null.

MDM Use Case — Score Drop Detection

After ranking, you want to know: is the rank-1 record clearly better than rank-2, or are they almost identical? A big score gap = confident match. A tiny gap = borderline, might need human review.

idclusternamescorerankprev_score (LAG)score_dropverdict
R1C1John Smith0.941null (no one above)—clear winner
R2C1Jon Smyth0.8120.940.13 dropconfident gap
R5C3Sarah Jones0.911null—borderline
R6C3Sara Jones0.9120.910.00 dropflag for review
PySpark — LAG for score drop detection
from pyspark.sql import Window from pyspark.sql import functions as F w = Window.partitionBy("cluster_id").orderBy(F.desc("score")) # LAG(column, n, default) — look n rows BACK in the window # For the first row (rank 1), there's no previous row → returns default (None) df = df.withColumn( "prev_score", F.lag("score", 1, None).over(w) # ↑col ↑how many rows back ↑default if no previous row ) # Compute how much the score dropped from the row above df = df.withColumn( "score_drop", F.col("prev_score") - F.col("score") # For rank-1 rows: prev_score is null → score_drop is null (not an error) ) # Flag clusters where rank-1 and rank-2 are almost identical scores # A tiny drop (< 0.05) means the "winner" isn't clearly better df = df.withColumn( "needs_review", F.when( (F.col("rank") == 2) & (F.col("score_drop") < 0.05), True ).otherwise(False) )
LEAD works the same way but looks forward. F.lead("score", 1, None).over(w) gives you the score of the row ranked one position below the current one. Useful when you're sitting on rank-1 and want to know how far ahead you are from rank-2 without needing to join back to the same table.

Stage 06

Running Aggregates — Count and Sum Without Collapsing

Window functions can also run aggregations — count, sum, avg — over the window, without collapsing rows. The result is added as a new column to every row. In MDM, this is how you count how many matches each record has, or sum up the scores of all records in a cluster.

Use Case — Match Count Per Record

After scoring, you want to know: how many times does each record appear in a matched pair? A record that matches many others is a strong candidate for the golden record. A record that matches nobody is probably unique.

Key observation: Every row in C3 gets cluster_size = 3 and cluster_avg_score = 0.823. The aggregation ran over the whole partition, and the result was broadcast back to every row — no collapse. This is the key difference from groupBy().count() which would give you one row per cluster instead of keeping the originals.
PySpark — running aggregates over a window
from pyspark.sql import Window from pyspark.sql import functions as F # For aggregates over the whole partition, you don't need orderBy # (You're not ranking — just counting/summing the whole group) w_agg = Window.partitionBy("cluster_id") # ↑ No .orderBy() needed for whole-partition aggregates # How many records are in this cluster? df = df.withColumn("cluster_size", F.count("*").over(w_agg)) # What is the average score in this cluster? df = df.withColumn("cluster_avg", F.avg("score").over(w_agg)) # What is the highest score in this cluster? df = df.withColumn("cluster_max", F.max("score").over(w_agg)) # All three columns added. Row count unchanged. Full records preserved. # Practical use: flag clusters where average score is low # (the whole cluster might be uncertain — send for manual review) df = df.withColumn("cluster_uncertain", F.col("cluster_avg") < 0.75 )
When NOT to use window aggregates: If you only need the aggregated number (not per-row context), groupBy().agg() is faster — it doesn't need to broadcast the result back to every row. Use window aggregates when you need the aggregated value alongside the original row data.

Stage 07

Real MDM Patterns

Everything in this module in one place — the complete golden record selection pipeline using window functions. This is what the post-scoring stage of your MDM pipeline looks like end to end.

Pattern 1 — Latest Record Per Entity (Source System Dedup)

Before entity resolution even starts, you often need to deduplicate within a single source system: if the CRM has 3 versions of the same customer (edited over time), keep only the latest.

Pattern 1 — keep latest record per entity per source
from pyspark.sql import Window from pyspark.sql import functions as F # Problem: CRM has 3 rows for customer_id=42 (3 edits over time). # We want the most recent one only. w_latest = Window\ .partitionBy("source_system", "customer_id")\ # ← group by source + customer .orderBy(F.desc("updated_at")) # ← most recent first df = df.withColumn("row_num", F.row_number().over(w_latest)) # Keep only the most recent version of each customer per source df_latest = df.filter(F.col("row_num") == 1).drop("row_num")

Pattern 2 — Full Golden Record Selection

After clustering and scoring, select one golden record per cluster, flag uncertain ones, and attach cluster metadata:

Pattern 2 — complete golden record pipeline
from pyspark.sql import Window from pyspark.sql import functions as F # ── Step 1: define window spec ────────────────────────────── # Primary sort: highest score first # Tie-breaker: most recent update wins w = Window\ .partitionBy("cluster_id")\ .orderBy(F.desc("score"), F.desc("updated_at")) # ── Step 2: rank within each cluster ─────────────────────── df = df.withColumn("rank", F.row_number().over(w)) # ── Step 3: score of the runner-up (for gap analysis) ────── # LAG looks 1 row back — for rank-2 row, this is the rank-1 score df = df.withColumn("prev_score", F.lag("score", 1).over(w)) df = df.withColumn("score_gap", F.col("prev_score") - F.col("score")) # ── Step 4: cluster-level aggregates ─────────────────────── w_agg = Window.partitionBy("cluster_id") # no orderBy for whole-partition agg df = df.withColumn("cluster_size", F.count("*").over(w_agg)) df = df.withColumn("cluster_avg", F.avg("score").over(w_agg)) # ── Step 5: select rank-1 records = golden records ───────── golden = df.filter(F.col("rank") == 1) # ── Step 6: flag uncertain clusters for human review ─────── # A cluster is uncertain if: avg score is low OR cluster is large golden = golden.withColumn("needs_review", (F.col("cluster_avg") < 0.75) | (F.col("cluster_size") > 5) ) # ── Step 7: clean up helper columns ──────────────────────── golden = golden.drop("rank", "prev_score") # Final result: one golden record per cluster, with: # - cluster_size: how many duplicates were merged # - cluster_avg: average confidence across the cluster # - score_gap: how much better this record scored vs runner-up # - needs_review: flag for uncertain clusters
