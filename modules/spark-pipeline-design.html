Stage 00 · Overview

You Built Every Part. Now Design the Machine.

Eight modules, each a single mechanism in isolation: standardize strings, block to dodge N², score fields, tune thresholds, find candidates with LSH, cluster pairs into entities, make it cheap. This capstone does the one thing none of them did — it assembles them into a single end-to-end architecture and asks the questions only the whole pipeline raises: where does data get written down between stages, how does the engine read its config, how do you turn cluster IDs back into master records, where does external data join in, what happens on tomorrow's run, and — the thread that runs through all of it — how does a decision at one stage ripple to the final output.

Every prior module, now one machine — the MDM entity-resolution architecture
Standardize→ Block→ LSH→ Score→ Classify→ Cluster→ Join-Back→ Golden Record

The first six boxes you already built (Modules 02–08). The capstone adds the last two — join-back and the golden record — and, more importantly, the connective tissue between every box: the intermediate datasets, the config engine, the orchestration, and the failure recovery that turn a pile of transforms into a pipeline you can run every night.

Stage 01 · The shape of the pipeline

The Pipeline Is a DAG, Not a Script

A script is a list of statements that run top to bottom. A pipeline is a directed acyclic graph: each stage is a node that consumes specific inputs and produces a specific output, and the edges are the datasets that flow between them. Thinking in the DAG — not the line-by-line script — is what lets you reason about what can run in parallel, what must run first, and what has to be rebuilt when one stage changes.

Every stage is a contract: inputs → output

The discipline that makes a pipeline maintainable is treating each stage as a black box with a typed contract. As long as a stage produces the columns the next stage expects, you can rewrite its internals freely. Here is the full MDM contract chain:

StageConsumesProducesModule
Standardizeraw customersstd: normalized fields02
Blockstdcandidates: pairs in same block00
LSHstdcandidates+: cross-block pairs06
Scorecandidates + stdscored: pair + similarity04
Classifyscoredverdicts: match / review / no05
Clusterverdicts (match edges)clusters: record → cluster_id07
Join-backclusters + stdmasters: one golden record / cluster09
Anchor — a build graph, not a Makefile recipe
You already think this way about builds. main.o depends on main.c; the linker depends on every .o. Change one .c and only its .o and the final link rebuild — not the whole tree. An ER pipeline is the same DAG: change the scoring stage and you must rebuild scored → verdicts → clusters → masters, but not std or the blocks. Knowing the DAG tells you exactly the blast radius of any change.

Why "acyclic" matters

The graph has no cycles: data flows forward, a stage never feeds back into one before it. That's what makes the pipeline re-runnable and reasoned about — you can topologically order the stages, know which are independent (blocking and LSH both read std and can run in parallel), and know that rebuilding from any node forward gives a correct result. A pipeline with a hidden cycle (a stage that reads its own future output) can't be safely re-run, and that's almost always a bug.

Two stages read the same input → they can run in parallel. Blocking and LSH both consume std and neither depends on the other; their outputs are unioned before scoring (Module 06's hybrid candidate set). Seeing that in the DAG is how you find free parallelism — and how Spark's scheduler finds it too.

Stage 02 · Materialization

Intermediate Datasets — Where You Write Things Down

Spark is lazy: chain ten transforms and nothing runs until an action forces the whole lineage at once. That's elegant for one query and a liability for a production pipeline, because a failure at stage 9 recomputes stages 1–8 from raw source, and you can't inspect what stage 4 produced because stage 4 never independently existed. The fix is deliberate materialization: write each stage's output to durable storage and let the next stage read it back.

One lazy DAG vs. materialized stages

✗ One lazy chain to the final write
Recompute-the-world on every failure

Standardize → block → LSH → score → classify → cluster → write, all as one lazy lineage. The cluster step fails at 2 a.m. on a bad config. The retry re-reads 2 TB and re-runs every stage from scratch — and you have no artifact to inspect to see which stage went wrong.

One bug = full recompute. Nothing is debuggable in isolation.

✓ Materialize between stages
Restart from the last good dataset

Each stage writes Parquet: std/, candidates/, scored/, verdicts/, clusters/, masters/. The cluster step fails — you fix the config and restart from verdicts/, skipping the expensive standardize-block-score chain entirely. And you can SELECT from scored/ to see exactly what the scorer produced.

One bug = rerun one stage. Every stage is an inspectable artifact.

The three things materialization buys you

Benefit 1
Re-runnability
Restart from the last completed stage instead of from raw source. Turns a 3-hour recovery into a 10-minute one.
Benefit 2
Debuggability
Every stage's output is a queryable table. "Why did these two merge?" → read scored/ and verdicts/ for that pair. No artifact, no answer.
Benefit 3
Idempotency
Write with mode("overwrite") keyed by run date. Re-running the same stage on the same input produces the same output — safe to retry without duplicating data.
The cost is real — don't materialize everything. Each write is a full pass to disk and each read pays it back. Materialize at meaningful boundaries (expensive stages, natural restart points, things you'll inspect), not after every withColumn. The art is choosing checkpoints: after standardize (expensive, reused), after scoring (the natural debug surface), after clustering (the hand-off to masters). Between those, let Spark's lazy chain do its job. This is the same lazy/recompute tension from Module 08's caching — materialization is caching that survives the job.
Checkpoint vs. cache vs. write. .cache() (Module 08) keeps a DataFrame in memory within one job. .checkpoint() writes it to reliable storage and truncates the lineage so a failure can't recompute past it. A plain .write.parquet() persists it as a named dataset the next job reads. A production pipeline uses the third between stages and the first two inside them.

Stage 03 · Configuration architecture

The YAML Plugin Engine — Behavior as Config, Not Code

Your real system isn't seven hard-coded functions; it's a PipelineEngine that runs a list of plugins, each driven by YAML. The blocking key, the field weights, the match thresholds — none of those are in the code. They're configuration the engine reads at runtime. This is the single most important architectural decision in the whole pipeline, because it decides whether tuning the system needs a code review and a deploy, or an edit to a config file.

Config vs. logic — the line that runs through every module

You've met this split in every scoring and threshold module. Here it becomes the architecture itself. The logic — how to compute a Jaro-Winkler similarity, how to run union-find — lives in plugin code and changes rarely. The config — which fields to compare, what weight each gets, where the thresholds sit — lives in YAML and changes constantly as you tune.

✗ Tuning lives in the code
Every threshold change is a deploy

The match threshold 0.72 is a literal in scoring.py. The data-quality team wants to try 0.75. That's a pull request, a review, a merge, a build, and a deploy — for one number. Nobody experiments, so the thresholds never get tuned.

✓ Tuning lives in YAML
Threshold change is a config edit

The threshold is scoring.thresholds.match: 0.72 in pipeline.yaml. The engine reads it at runtime. Trying 0.75 is a one-line edit and a re-run — no deploy. The people who understand the data can tune it directly.

What the engine reads

pipeline.yaml — the whole pipeline as configuration

# Each plugin's behavior is data the engine reads — not code it runs.
standardize:
  nickname_expansion: true          # Bob → Robert
  street_abbreviations: true        # St → Street
blocking:
  key: "zip + last_name[0]"        # the blocking strategy
  max_block_size: 5000             # skew guard (Module 00)
lsh:
  enabled: true
  bands: 6
  rows: 4                          # S-curve threshold (Module 06)
scoring:
  weights: {name: 0.40, address: 0.25, dob: 0.20, email: 0.15}
  null_handling: "renormalize"      # drop absent fields (Module 04)
  thresholds: {match: 0.72, review: 0.55}
clustering:
  algorithm: "connected_components"
  max_cluster_size: 50            # bridge-edge guard (Module 07)

Read that YAML again with the module numbers in mind: every tunable you learned across the track is a key here. The engine's job is to read this file, instantiate each plugin with its slice of config, and run them in DAG order. The plugins are the logic; this file is the behavior.

This is exactly what the Stage 10 simulator exposes. Every control in the flagship simulator maps to one of these YAML keys — and the simulator labels each control with the key it drives. When you drag the match-threshold slider, you're editing scoring.thresholds.match; when you switch the blocking key, you're editing blocking.key. The simulator is this config file, made live.

Stage 04 · Assembly

Join-Back — Turning cluster_id Into a Master Record

Clustering (Module 07) doesn't produce master entities. It produces a thin mapping: record_id → cluster_id. Every record now knows which entity it belongs to, but the actual golden record — the single best name, the current address, the canonical email — hasn't been built yet. That's the join-back: take the cluster assignment, join it back to the full standardized records, and collapse each cluster into one survivorship-resolved master.

The two-step assembly

First, join the skinny clusters table (record_id, cluster_id) back to the wide std table to attach every field to its cluster. Then, within each cluster_id, pick the surviving value per field — most-recent address, most-complete name, any non-null email. That second step is Module 03's window functions doing exactly what they were built for: row_number() over a partition by cluster_id, ordered by recency or completeness.

join-back + survivorship → masters

from pyspark.sql import functions as F, Window

# 1. attach every field to its cluster — join skinny clusters → wide std
enriched = clusters.join(std, "record_id")

# 2. survivorship: latest record wins per cluster (Module 03 window)
w = Window.partitionBy("cluster_id").orderBy(F.col("updated_at").desc())
masters = (enriched
    .withColumn("rn", F.row_number().over(w))
    .filter(F.col("rn") == 1)          # one golden record per cluster
    .drop("rn"))

Broadcast vs. shuffle join-back — the cost decision

The join-back is a real join, and which join strategy it uses decides its cost (Modules 00 & 08). The deciding question is the size of the clusters side:

SituationJoin strategyWhy
clusters mapping fits in memory (millions of rows, 2 cols)broadcastship the skinny map to every executor; no shuffle of the wide std
clusters too large to broadcastsort-mergeboth sides shuffle on record_id; unavoidable at that scale
The usual MDM casebroadcast the maprecord_id → cluster_id is two narrow columns — almost always broadcastable even for big inputs
The skinny side is the broadcast side. The clusters mapping is just two columns, so even a hundred million records is a few GB — often broadcastable, and AQE may switch to broadcast on its own (Module 08). The win is large: broadcasting the map means the wide std table never shuffles. Wrap it explicitly — F.broadcast(clusters) — when you want certainty rather than hoping the optimizer notices.
Survivorship is a policy, not a default. "Latest record wins" is one rule; "most-complete record wins," "trust source A over source B," and "most-frequent value wins" are others — and they produce different golden records from the same cluster. The simulator's golden-record control lets you flip between most-recent and most-complete and watch master quality move. Picking the wrong survivorship rule silently corrupts every master, with no error to warn you.

Stage 05 · External data

Enrichment — Joining in the Outside World

Real MDM doesn't resolve in a vacuum. You join in external reference data: a postal service that validates and canonicalizes addresses, a third-party file that adds firmographics, a phone-validation service, a watchlist. Enrichment is just more joins — but where in the DAG you place them changes both the result and the cost, and getting the placement wrong is a classic design smell.

Where enrichment belongs in the DAG

EnrichmentPlace it…Why there
Address validation / canonicalizationbefore blockingit improves the match signal — a canonical address blocks and scores better. Enrich early so every downstream stage benefits.
Firmographic / demographic appendafter clusteringit doesn't help matching; append it to the finished master so you enrich one row per entity, not every duplicate.
Watchlist / compliance flagon the mastera property of the resolved entity, not the raw record — compute it once on the golden record.
✗ Enrich after clustering — when it was a match signal
You threw away recall

You append validated addresses after clustering. But a canonical address would have made two records of the same person block together and score higher. By enriching late, the matcher never saw the better signal — so those records never merged, and now you're enriching two separate masters that should have been one.

✓ Enrich before blocking — when it's a match signal
The signal helps every downstream stage

Validate addresses first. Now "12 Oak St" and "12 Oak Street, Apt 0" both canonicalize to the same string — they block together, score high, and merge. Enrichment that improves matching must happen upstream of matching, or its value is lost.

The mechanism: broadcast the reference table

Reference data is usually small relative to your records — a postal lookup, a country-code table, a few million firmographic rows. That makes it the textbook broadcast-join case from Module 08: ship the small reference table to every executor and join without shuffling your big record set.

broadcast enrichment — small reference, big records, no shuffle

ref = spark.read.parquet("s3://ref/postal_canonical")   # ~small
std_enriched = std.join(F.broadcast(ref), "raw_address", "left")
# every executor gets the postal table; std never shuffles. Enrich, THEN block.
External joins can explode rows. A reference table with multiple matches per key (a non-unique join key) silently multiplies your record count — and downstream that means duplicate candidate pairs and skewed scores. Always confirm the enrichment join is one-to-one (or deliberately one-to-many) before trusting it; a fan-out enrichment is a correctness bug that looks like a performance bug.

Stage 06 · Running it tomorrow

Incremental vs. Full Reprocessing

Everything so far described resolving the whole dataset once. But you run this pipeline every night, and 99% of the data didn't change. Re-resolving 2 TB from scratch to absorb 50 000 new records is the most common waste in production MDM — and yet "just match the new records against the existing masters" hides a genuinely hard problem. This stage is the design choice between the two.

The two strategies

Full reprocessing
Re-resolve everything, every run

Read all records, run the whole DAG, rebuild all masters. Correct and simple — every run sees the global picture, so clustering is always optimal. Expensive — you pay the full N² blocking and scoring cost nightly even though almost nothing changed.

Use when: data is small enough, or correctness matters more than cost, or the match logic just changed (you want a full re-resolve).

Incremental resolution
Match only new/changed records against existing masters

Block and score the day's new records against the existing master set, attach them to a master or spin up a new one. Cheap — cost scales with the delta, not the corpus. Subtle — a new record can be the bridge that should merge two previously-separate masters, and a naive incremental run misses it.

Use when: the corpus is large and the daily delta is small — the usual production case.

Why incremental is genuinely hard — the bridge problem

This is the same transitive-closure fact from Module 07, now across time. Masters A and B exist separately because nothing ever linked them. Tonight's new record C matches both A and B. Correctly, C should merge A and B into one master. A naive incremental run that only asks "which existing master does C join?" picks one and silently leaves a duplicate. Handling bridges means incremental resolution must be able to merge existing masters, not just attach new records — which is most of its complexity.

The drift trap. Incremental runs accumulate small errors — a slightly-wrong early merge stays wrong forever because you never re-examine it. The standard mitigation is a periodic full re-resolve (e.g. nightly incremental, weekly full) to wash out drift, plus an unconditional full re-resolve whenever the match logic or thresholds change — because new logic invalidates every past incremental decision. Incremental for speed, full for correctness, on a schedule.
Changing the YAML invalidates incremental state. If you bump scoring.thresholds.match from 0.72 to 0.78, every master built under the old threshold is now suspect — some pairs that merged wouldn't merge under the new rule. You cannot incrementally patch that; it demands a full re-resolve. Tie your config version to your master version, and force a full run on any config change that touches matching.

Stage 07 · Running it reliably

Orchestration & Failure Recovery

A pipeline that only works when every stage succeeds on the first try isn't a production pipeline. Orchestration is the layer that runs the stages in dependency order, retries the ones that fail, and — critically — knows which stages are safe to retry. The materialized intermediate datasets from Stage 02 are what make all of this possible; orchestration is the payoff for having written things down.

The orchestration job: order, retry, recover

Order
Topological execution
Run stages in DAG order; run independent stages (blocking, LSH) in parallel. The orchestrator (Airflow, Dagster, a DAG runner) reads the dependency graph and schedules it.
Retry
Restart from last good dataset
A stage fails → don't restart the pipeline. Restart from the last materialized output. The scored/ dataset survived; rerun classify onward only.
Recover
Idempotent, safe-to-retry stages
A retried stage must produce the same result and not double-write. overwrite keyed by run-id makes a stage safe to run twice — the second run replaces, never appends.

Which stages are safe to re-run?

A stage is safe to retry when it's idempotent: same input → same output, no side effect that accumulates. Pure transforms that overwrite their output are always safe. The dangerous ones are stages with external side effects — appending to a queue, calling an API with a usage cost, sending notifications — because a retry repeats the side effect.

StageSafe to retry?Why
Standardize / score / cluster (overwrite output)yespure transform; rerun replaces the dataset
Join-back / masters (overwrite)yesdeterministic from clusters + std
Push to review queue (append)not naivelya retry double-appends; needs dedupe-on-key or overwrite-by-run
Notify / call paid enrichment APInoexternal side effect repeats; needs an idempotency key
Checkpoint placement is an orchestration decision. You checkpoint (Stage 02) at the boundaries the orchestrator will restart from. Put a materialization boundary after every stage you'd hate to recompute — standardize (expensive), scored (the debug surface), clusters (the hand-off). The orchestrator's retry granularity can be no finer than your materialization granularity: you can only restart from a dataset that exists.

Stage 08 · Watching the pipeline

Observability — Every Stage Emits Its Own Metric

A pipeline you can't see is a pipeline you can't trust. But the mistake is watching one global number — "recall" — and assuming it tells you where a problem is. It doesn't. Each stage has a different governing metric, and a drop in the final number could originate at any stage. Good observability means each stage emits the metric that actually governs it, so when the system degrades you can read which stage moved.

The metric that governs each stage

This is the single most important idea for the simulator that follows. Precision and recall are system outcomes — they fall out at the end. But each stage is governed by its own local metric, and that local metric is what you tune the stage against:

StageIts governing metricWhat it tells you
Standardizecoverage% of true duplicates the normalization made comparable at all
Blockreduction ratio & pair completenesshow many comparisons you saved vs. how many true pairs you kept
LSHpairs recoveredtrue cross-block pairs caught vs. false candidates added
Scoreseparationthe gap between match scores and non-match scores — can you even draw a line?
Classifyprecision / recall / queuewhere the thresholds land the three-way split
Clustercluster count & over-mergeshow many masters, and how many fuse different people
Golden recordcompletenesshow filled-in the surviving master is
Anchor — a factory line, gauge per station
A factory doesn't watch only the count of finished cars. Each station has its own gauge — paint thickness here, torque there, alignment at the next. When the finished count drops, you don't guess; you walk the line reading gauges until one is out of band. The pipeline is the same: separation is the scoring station's gauge, reduction ratio is blocking's. Watch all of them, and a degraded final number points at itself.
This is what the simulator badges. In Stage 10, every stage node shows its own governing metric live. When you turn a knob and the final master count moves, you can walk the flow and see which stage's gauge moved first — exactly the observability discipline, made visible.

Stage 09 · The climax

Recall, Precision, Cost: System Properties, Not Stage Properties

Here is the one idea the whole capstone is built to deliver. Every prior module taught you to tune a stage. This stage tells you that you can't — not in isolation. A knob at blocking changes the final master count. A weight at scoring changes the review-queue size. The pipeline's recall, precision, cost, and review burden don't belong to any stage; they emerge from the composition of every stage's setting. Tuning one stage while staring at one stage's metric is the central beginner mistake — and the most expensive one.

✗ Red herring — tune the stage in isolation
"Recall is low, so loosen the scorer"

You lower the match threshold to catch more pairs. Recall barely moves — because the missing pairs were never candidates; blocking dropped them two stages upstream. Meanwhile precision craters and the review queue overflows, because the looser threshold fused records that were candidates but shouldn't match. You tuned the wrong stage and damaged two others.

✓ What actually decides it — the system response
Trace the ripple to find the real lever

Low recall with the missing pairs never reaching the scorer means the lever is blocking (or LSH), not the threshold. Change the blocking key, watch recall climb and cost climb — and accept that tradeoff knowingly. The right move is the one you choose after seeing the whole-system response, not the local one.

The couplings you'll feel in the simulator

Turn this knob……and this moves downstream
Normalization none → aggressivenickname duplicates become comparable → recall up, no cost change
Blocking key broader (soundex)moved-person pairs co-block → recall up, comparisons up (cost up)
Enable LSHcross-block true pairs recovered → recall up, and some false candidates added
Lower match thresholdrecall up, precision down, review queue grows, over-merges appear
Transitive clustering offthe moved person stays split → more masters, lower recall on entities
Max-cluster guard tighterover-merges blocked → precision up, but a real large cluster may split
Tuning a stage in isolation is a red herring; the system response decides. There is no "correct" setting for blocking, or scoring, or clustering on its own. There is only a correct system — a combination that hits your target recall, precision, cost, and review capacity together. The next stage lets you hold the whole system in your hands and feel every one of these couplings at once.

Stage 11 · The production pattern

The PipelineEngine — Plugins in DAG Order

Here is the architecture from Stage 03 as a skeleton: an engine that reads the YAML, instantiates each plugin with its config, and runs them in order — materializing between stages (Stage 02) so the run is restartable (Stage 07). This is the spine every prior module's logic plugs into.

pipeline_engine.py — config-driven, materialized, restartable

import yaml
from pyspark.sql import functions as F

class PipelineEngine:
    def __init__(self, config_path, run_id):
        self.cfg    = yaml.safe_load(open(config_path))   # behavior = config
        self.run_id = run_id
        self.base   = f"s3://mdm/runs/{run_id}"

    def _stage(self, name, fn, *inputs):
        """Run one stage, materialize its output, make it restartable."""
        out = f"{self.base}/{name}"
        if self._exists(out):                       # already done → skip (restart)
            return spark.read.parquet(out)
        df = fn(*inputs, self.cfg)                   # plugin logic + its config
        df.write.mode("overwrite").parquet(out)  # idempotent materialize
        return spark.read.parquet(out)

    def run(self):
        std        = self._stage("std",        standardize, raw)
        blk        = self._stage("candidates", block,       std)   # both read std →
        lsh        = self._stage("lsh",        lsh_pairs,   std)   # independent, parallel
        cands      = blk.union(lsh).distinct()              # hybrid set (Mod 06)
        scored     = self._stage("scored",     score,    cands, std)
        verdicts   = self._stage("verdicts",   classify, scored)
        clusters   = self._stage("clusters",   cluster,  verdicts)
        masters    = self._stage("masters",    join_back, clusters, std)  # Stage 04
        return masters

Read the shape, not the syntax. Three things make this an engine rather than a script: behavior comes from self.cfg (config, not literals — Stage 03); _stage materializes every output and skips any that already exists (restartable — Stages 02 & 07); and the stages compose in DAG order with blocking and LSH visibly independent (Stage 01). Swap any plugin's internals and nothing else changes — the contracts hold.
The skip-if-exists line is the restart logic. Because each stage checks for its output before computing, a re-run after a failure replays instantly through every completed stage and resumes at the first missing one. That single if self._exists(out) is what turns "rerun the whole pipeline" into "resume where it died" — but it only works because every stage materialized. Architecture is the payoff for the boring discipline of writing things down.

Stage 12 · Decision criteria

Architecture Decision Tree & Design Smells

Designing the pipeline is a sequence of choices, each with a clear deciding question. Run these top to bottom when you're laying out a new ER pipeline — then check your design against the anti-patterns, which are the same choices made wrong.

Decision 1
Full re-resolve or incremental?
→ Delta size decides. Small corpus or correctness-critical or match-logic just changed → full. Large corpus, small daily delta → incremental, with a periodic full re-resolve to wash out drift. Never run incremental across a config change that touches matching.
Decision 2
Where do you materialize?
→ At expensive stages and restart points. After standardize (expensive, reused), after scored (the debug surface), after clusters (the hand-off). Not after every transform — each checkpoint is a full disk pass. Your retry granularity equals your materialization granularity.
Decision 3
Broadcast or shuffle the join-back?
→ Size of the skinny side. The record_id → cluster_id map is two columns — almost always broadcastable, even at hundreds of millions of rows. Broadcast it so the wide std never shuffles. Fall back to sort-merge only when the map genuinely won't fit.
Decision 4
Where does enrichment go?
→ Is it a match signal? Yes (address validation) → before blocking, so matching benefits. No (firmographics, flags) → after clustering, on the master, so you enrich one row per entity instead of every duplicate.
Decision 5
Add LSH, or change the blocking key?
→ Both recover cross-block pairs; pick by cost & control. A broader blocking key (soundex) can recover the moved person without LSH but raises comparisons across the board. LSH targets just the cross-block pairs above a tunable threshold. If one blocking key fixes it cheaply, prefer it; reach for LSH when no single key covers your match patterns.

Honest Gaps
What this capstone didn't cover
Real incremental implementation: the bridge-merge problem was explained but not built — production incremental ER (delta detection, master-merge logic, state management) is its own large topic.

Streaming / real-time ER: everything here is batch. Resolving entities on a stream (Structured Streaming, stateful matching, late data) is a different architecture.

Survivorship depth: field-level survivorship rules, source-trust hierarchies, and conflict resolution were sketched, not drilled.

Data quality & lineage tooling: the observability stage named per-stage metrics but didn't cover expectation frameworks (Great Expectations), data contracts enforcement, or column-level lineage.

The simulator is a teaching model: 24 records with simplified deterministic sims. Real similarity (Jaro-Winkler, learned weights), real LSH (MinHash signatures), and real cluster sizes behave proportionally but not identically — the couplings are faithful, the exact numbers are illustrative.

Governance & serving: how masters are versioned, served to consumers (an API, a feature store), and audited for compliance is the layer above this pipeline.
