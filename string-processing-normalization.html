Stage 01

Why Normalize First

Similarity algorithms measure string distance. If "John Smith" and "john smith" score 0 because of case, you've wasted the scoring stage entirely. Normalization collapses surface variation so algorithms measure semantic distance — not formatting noise.

The Core Problem: Same Entity, Different Strings

Your pipeline receives records from multiple source systems — CRM, ERP, billing, support — each with its own formatting conventions. These are all the same person:

📁
CRM System
JOHN A. SMITH-JONES
📁
ERP System
Smith-Jones, John
📁
Billing System
J. Smith Jones
📁
Support Ticket
john smithjones

Without normalization, Levenshtein distance between "JOHN A. SMITH-JONES" and "john smithjones" is 8 edits — which looks like two very different entities. After normalization both become john smithjones — distance drops to 0.

The rule: Normalization is not about making strings "pretty." It's about ensuring that surface variation that carries no semantic meaning (case, punctuation, whitespace) does not consume your edit budget.

What Normalization Fixes vs What Scoring Handles

These are two different jobs. Mixing them up leads to either over-normalizing (losing real signal) or under-normalizing (wasting scoring budget on noise).

Variation TypeExampleFix WhereWhy
Case differenceJOHN vs johnNormalizationNo semantic difference, pure formatting
PunctuationSmith-Jones vs Smith JonesNormalizationHyphen carries no identity signal
WhitespaceJohn Smith vs John SmithNormalizationExtra spaces are data entry noise
AbbreviationSt. vs StreetNormalizationKnown equivalence, deterministic mapping
Typo / transpositionJohn vs JonScoringGenuine variation — needs fuzzy algorithm
NicknameBill vs WilliamScoringSemantic equivalence — lookup table or model
Missing middle nameJohn A Smith vs John SmithScoringStructural difference — scoring handles weight
Completely different nameJohn Smith vs Sarah JonesNeither — not a matchNo normalization or scoring rescues this
Common trap: Teams add nickname expansion (Bill→William) to the normalization plugin. This loses the signal that the original said "Bill" — which might matter for confidence scoring. Keep deterministic equivalences in normalization, probabilistic ones in scoring.

Stage 02

The Normalization Pipeline

Normalization is a sequence of deterministic transformations — each step shrinks the variation space. Order matters: lowercase before stripping so you don't accidentally lose case-sensitive abbreviations; strip before tokenizing so you split on clean tokens.

Step-by-Step: Tracing a Real Name

Watch "Dr. PRIYA PATEL-SHARMA, PhD" flow through each stage of your StandardizationPlugin:

1
Lowercase
"Dr. PRIYA PATEL-SHARMA, PhD"
↓
"dr. priya patel-sharma, phd"
2
Strip titles & suffixes (config-driven)
"dr. priya patel-sharma, phd"
↓
"priya patel-sharma"
3
Replace punctuation with space
"priya patel-sharma"
↓
"priya patel sharma"
4
Collapse whitespace
"priya patel sharma"
↓
"priya patel sharma"
5
Sort tokens (optional — for name reordering)
"priya patel sharma"
↓
"patel priya sharma" ← tokens sorted alphabetically
Token sorting is a double-edged sword. It lets "Smith John" match "John Smith" — but it also makes "John Williams" match "Williams John" (a different person with a transposed first/last). Only use token sort when your source data has known first/last name ordering inconsistencies. Don't apply it globally.

Your StandardizationPlugin in PySpark

This maps directly to what your plugin YAML config controls. Each step below corresponds to a plugin config key:

PySpark — StandardizationPlugin transform method
from pyspark.sql import functions as F import re def normalize_name(df, col_name): # Step 1: Lowercase df = df.withColumn(col_name, F.lower(F.col(col_name))) # Step 2: Strip titles/suffixes (your YAML config list) titles = ["dr", "mr", "mrs", "ms", "phd", "jr", "sr", "md"] pattern = r'\b(' + '|'.join(titles) + r')\.?\b' df = df.withColumn(col_name, F.regexp_replace(F.col(col_name), pattern, "")) # Step 3: Replace punctuation with space df = df.withColumn(col_name, F.regexp_replace(F.col(col_name), r"[^a-z0-9\s]", " ")) # Step 4: Collapse whitespace + trim df = df.withColumn(col_name, F.trim(F.regexp_replace(F.col(col_name), r"\s+", " "))) return df def normalize_phone(df, col_name): # Strip all non-digits df = df.withColumn(col_name, F.regexp_replace(F.col(col_name), r"[^\d]", "")) # Strip leading country code (1 for US) df = df.withColumn(col_name, F.when(F.length(F.col(col_name)) == 11, F.substring(F.col(col_name), 2, 10)) .otherwise(F.col(col_name))) return df def normalize_email(df, col_name): # Lowercase + trim only — email is case-insensitive by spec df = df.withColumn(col_name, F.trim(F.lower(F.col(col_name)))) return df

Stage 03

Levenshtein Distance

The foundation of all edit-based similarity. Levenshtein distance counts the minimum number of single-character operations — insert, delete, substitute — to transform string A into string B. It's the "how far apart are these strings?" question answered as an integer.

The Three Operations

Insert
Add a character
color → colour
Insert 'u' at position 5. Cost: 1.
Delete
Remove a character
colour → color
Delete 'u' at position 5. Cost: 1.
Substitute
Replace a character
Jon → John
Wait — that's actually an insert, not a sub. Levenshtein finds the minimum path.

Step-by-Step: "kitten" → "sitting" — Read This First

1
What does the matrix represent?
Rows = characters of A ("kitten"), Columns = characters of B ("sitting"). The "" row/column = the empty string.
→
Cell [i][j] = minimum edits to turn the first i characters of A into the first j characters of B. The bottom-right cell [6][7] = the final answer.
2
Why does column 0 go 0, 1, 2, 3, 4, 5, 6?
Column 0 = "how many edits to turn the first i chars of A into the EMPTY string?"
→
You need i deletions. "k"→"" = 1 delete. "ki"→"" = 2 deletes. "kit"→"" = 3 deletes. Column 0 is just the row index.
3
Why does row 0 go 0, 1, 2, 3, 4, 5, 6, 7?
Row 0 = "how many edits to build the first j chars of B from the EMPTY string?"
→
You need j insertions. ""→"s" = 1 insert. ""→"si" = 2 inserts. Row 0 is just the column index.
★
The rule for every other cell — two cases:
CASE 1: A[i] == B[j] (characters match) → copy the diagonal cell above-left. No edit needed.
→
CASE 2: A[i] != B[j] (no match) → 1 + minimum of: (cell above = delete), (cell left = insert), (cell diagonal = substitute). Pick the cheapest.
?
Why is the answer 3? What are the 3 edits?
kitten → sitten (substitute k→s) → sittin (substitute e→i) → sitting (insert g at end)
→
3 edits. Bottom-right cell = 3. Similarity = 1 − 3/7 = 0.571. Not a great match — which is correct, "kitten" and "sitting" are genuinely different.

Normalized Levenshtein → Similarity Score

Raw Levenshtein distance is an integer — not directly comparable across different-length strings. "kitten"→"sitting" is distance 3 out of a max of 7, which is very different from distance 3 between two 4-letter words. Always normalize:

Normalized Levenshtein — Python + PySpark
# Normalized Levenshtein: 0.0 = totally different, 1.0 = identical def lev_similarity(a, b): dist = levenshtein_distance(a, b) max_len = max(len(a), len(b)) if max_len == 0: return 1.0 return 1.0 - (dist / max_len) # PySpark built-in from pyspark.sql.functions import levenshtein pairs = pairs.withColumn("name_lev_dist", levenshtein(F.col("a.name_std"), F.col("b.name_std"))) # Normalize to similarity score pairs = pairs.withColumn("name_lev_sim", 1.0 - (F.col("name_lev_dist") / F.greatest( F.length(F.col("a.name_std")), F.length(F.col("b.name_std")) )))

Where Levenshtein Falls Short for Names

Levenshtein treats every character position equally. For names, this causes two specific failures:

PairLev DistanceLev SimilarityHuman JudgmentProblem
Jon vs John 1 0.75 Very likely same Underscore — names that differ by 1 char are often the same person
Johnson vs Johnston 2 0.75 Maybe same OK — similar score, similar confidence
Mary vs Gary 1 0.75 Very unlikely same Overscore — a 1-char prefix difference changes identity entirely
Jonathan vs Johnathan 1 0.88 Almost certainly same Works well here — common variant spelling
The key insight: Levenshtein doesn't know that prefixes matter more than suffixes for names, or that transpositions ("Jon"/"Jno") are more likely data-entry errors than arbitrary substitutions. That's exactly what Jaro-Winkler was designed to fix.

Stage 04

Jaro-Winkler Similarity

Designed specifically for short strings and personal names. Unlike Levenshtein which counts edits, Jaro measures how many characters match within a proximity window, then penalizes transpositions. Winkler adds a prefix bonus — because names that share a prefix are much more likely to be the same person than names that only match in the middle.

Building Up Jaro: Three Components

Step 1
Find matching characters
A character in A "matches" a character in B if they're the same AND within a match window of floor(max(|A|,|B|)/2) - 1 positions. This handles transpositions.
Step 2
Count transpositions
A transposition is when a matching character appears in a different order between A and B. Transpositions count as half an error — less than a full mismatch.
Step 3 (Winkler)
Add prefix bonus
If A and B share the same prefix (up to 4 chars), the Winkler boost increases the score. p = 0.1 (standard). This is why "John"/"Johnathan" scores higher than "John"/"Bjohn".

The Jaro Formula — Worked Example

Let's compute Jaro similarity for "MARTHA" vs "MARHTA" (a transposition pair).

1
Match window
max(6,6)/2 - 1 = 2. A character must be within 2 positions.
2
Find matches (m)
MARTHA → M A R T H A
MARHTA → M A R H T A → m = 6 (all match, just reordered)
3
Count transpositions (t)
Matching sequence in MARTHA: M A R T H A
Matching sequence in MARHTA: M A R H T A → T and H swapped = 2 mismatches → t = 2/2 = 1
4
Jaro formula: ⅓ × (m/|s1| + m/|s2| + (m-t)/m)
⅓ × (6/6 + 6/6 + (6-1)/6) = ⅓ × (1 + 1 + 0.833) = 0.944
5
Winkler boost — shared prefix "MAR" (l=3)
jaro_winkler = 0.944 + (3 × 0.1 × (1 - 0.944)) = 0.944 + 0.017 = 0.961

Jaro-Winkler in Your ScoringPlugin

PySpark UDF — Jaro-Winkler scoring
from pyspark.sql import functions as F from pyspark.sql.types import FloatType import jellyfish # pip install jellyfish # Register as UDF jaro_winkler_udf = F.udf( lambda a, b: float(jellyfish.jaro_winkler_similarity(a, b)) if a and b else 0.0, FloatType() ) # Apply in your scoring plugin pairs = pairs.withColumn( "name_jw_sim", jaro_winkler_udf(F.col("a.name_std"), F.col("b.name_std")) ) # Weight into composite score pairs = pairs.withColumn( "name_score", F.col("name_jw_sim") * 0.7 + # Jaro-Winkler carries most weight F.col("name_lev_sim") * 0.3 # Levenshtein as secondary signal )
UDF performance note: Python UDFs break Spark's Catalyst optimizer — each row crosses the JVM→Python boundary. For production, use Spark's built-in levenshtein() for edit distance. For Jaro-Winkler, either use a Scala UDF or batch the computation with pandas_udf (vectorized). The naive UDF above will be 10–50× slower on large candidate pair sets.

Stage 05

Phonetic Encoding

Jaro-Winkler catches spelling variations of the same name. Phonetic encoding catches pronunciation variations — "Smith" and "Smyth" sound identical but have edit distance 1. These encodings are used for blocking, not scoring: they generate a key that groups sound-alike names into the same partition before the self-join.

Soundex — The Classic Algorithm

Soundex was developed in the 1910s for the US Census. It converts a name to a 4-character code: first letter + 3 digits representing consonant groups. All vowels and H, W, Y are ignored after the first character.

→
Soundex encoding of "SMITH"
S - M - I - T - H
S → keep first letter = S
M → group 5 = 5
I → vowel, ignored
T → group 3 = 3
H → ignored
Pad to 4 chars → S530
→
Soundex encoding of "SMYTH"
S - M - Y - T - H
S → S
M → 5
Y → ignored (like vowel in Soundex)
T → 3
H → ignored → S530 ← same code!

Real MDM Examples — Names That Soundex Groups Together

Soundex vs Double Metaphone

Soundex is simple and fast but has known weaknesses — particularly with non-English names and names starting with vowels. Double Metaphone handles a broader set of language patterns.

AlgorithmOutputHandlesMissesBest For
Soundex 1 letter + 3 digits English names, simple variants Non-English origins, vowel-start names Quick baseline, US data
Double Metaphone 2 codes (primary + alternate) Multi-language, Slavic, Hebrew, Asian romanizations Highly complex variants Diverse international MDM datasets
NYSIIS Variable length string Better vowel handling than Soundex Less commonly supported in libraries New York state records, genealogy

Adding Phonetic Blocking to Your Pipeline

PySpark — generating Soundex block keys
from pyspark.sql.functions import soundex, col # Add soundex block key on last name df = df.withColumn( "block_soundex_last", soundex(col("last_name_std")) ) # Multi-key blocking: soundex OR exact phone # explode into one row per block key from pyspark.sql.functions import array, explode, struct df_blocked = df.withColumn( "block_keys", array( struct(col("block_soundex_last").alias("key")), struct(col("phone_std").alias("key")) # exact phone as block key too ) ).withColumn("block_key", explode("block_keys")."key") # Deduplicate before self-join pairs = df_blocked.alias("a").join( df_blocked.alias("b"), (col("a.block_key") == col("b.block_key")) & (col("a.id") < col("b.id")) # a.id < b.id eliminates (A,B) + (B,A) duplicates ).dropDuplicates(["a.id", "b.id"])
Why multi-key blocking? A single Soundex key misses pairs that share an exact phone number but have very different names (e.g., family members). Multi-key blocking casts a wider net — then scoring filters the false positives. More candidate pairs = higher recall; better scoring = higher precision.

Stage 06

Choosing the Right Algorithm

No single algorithm wins everywhere. The right choice depends on field type, string length, data origin, and whether you're blocking or scoring. Here's the decision framework used in production MDM pipelines.

The Decision Framework

Q1
Are you blocking or scoring?
Blocking → use phonetic encoding (Soundex / Double Metaphone)
Scoring → continue to Q2
Q2
Is it a personal name (first or last)?
Yes → Jaro-Winkler (prefix-sensitive, transposition-aware)
No → continue to Q3
Q3
Is the string short (<20 chars) or long?
Short (email username, code, ID) → Normalized Levenshtein
Long (address, company name, description) → Token-based (Jaccard or cosine on token sets)
Q4
Is the field a structured format (phone, email, date)?
Yes → exact match after normalization (similarity is binary after good normalization)
No → apply the algorithm from Q2/Q3

Your ScoringPlugin — Putting It Together

In your MDM pipeline, the scoring plugin computes a weighted composite. Here's a realistic config-to-code mapping:

Scoring config (YAML) → PySpark implementation
# YAML plugin config # scoring: # fields: # - name: full_name, algorithm: jaro_winkler, weight: 0.40 # - name: email, algorithm: exact, weight: 0.30 # - name: phone, algorithm: exact, weight: 0.20 # - name: date_of_birth,algorithm: exact, weight: 0.10 # thresholds: # match: 0.85 # review: 0.70 from pyspark.sql import functions as F def score_pairs(pairs): # Name: Jaro-Winkler (via UDF or pandas_udf for performance) pairs = pairs.withColumn("s_name", jaro_winkler_udf(F.col("a.full_name_std"), F.col("b.full_name_std"))) # Email: exact after normalization → binary 0.0 or 1.0 pairs = pairs.withColumn("s_email", F.when(F.col("a.email_std") == F.col("b.email_std"), 1.0) .when(F.col("a.email_std").isNull() | F.col("b.email_std").isNull(), None) .otherwise(0.0)) # Phone: exact after normalization pairs = pairs.withColumn("s_phone", F.when(F.col("a.phone_std") == F.col("b.phone_std"), 1.0) .when(F.col("a.phone_std").isNull() | F.col("b.phone_std").isNull(), None) .otherwise(0.0)) # DOB: exact pairs = pairs.withColumn("s_dob", F.when(F.col("a.dob") == F.col("b.dob"), 1.0) .when(F.col("a.dob").isNull() | F.col("b.dob").isNull(), None) .otherwise(0.0)) # Weighted composite — skip null fields (don't penalize missing data) pairs = pairs.withColumn("composite_score", (F.coalesce(F.col("s_name"), 0.0) * 0.40 + F.coalesce(F.col("s_email"), 0.0) * 0.30 + F.coalesce(F.col("s_phone"), 0.0) * 0.20 + F.coalesce(F.col("s_dob"), 0.0) * 0.10)) # Classify return pairs.withColumn("verdict", F.when(F.col("composite_score") >= 0.85, "MATCH") .when(F.col("composite_score") >= 0.70, "REVIEW") .otherwise("NO_MATCH"))
Null handling trap: If a field is null for one record, using coalesce(score, 0.0) penalizes the pair for missing data — a record with no phone number scores 0 on phone, dragging down the composite. Instead, re-weight on the fly: compute the composite only over available fields and renormalize weights to sum to 1.0 among non-null pairs. This is what enterprise MDM platforms do under the hood.
